01

Git ও GitHub-এর পরিচিতি

🔧 Git-এর জন্ম

Linux অপারেটিং সিস্টেমের সোর্স কোড ম্যানেজ করার জন্য — কে, কখন এবং কী পরিবর্তন করেছে তা ট্র্যাক করার উদ্দেশ্যে Linus Torvalds ২০০৫ সালের এপ্রিলে মাত্র কয়েকদিনের মধ্যে Git তৈরি করেন। আজ এটি পৃথিবীর সবচেয়ে জনপ্রিয় ভার্সন কন্ট্রোল সিস্টেম।

🌐 DVCS — ডিস্ট্রিবিউটেড ভার্সন কন্ট্রোল সিস্টেম

❌ সেন্ট্রালাইজড সিস্টেম

ইন্টারনেট না থাকলে বা মূল সার্ভার ডাউন হলে পুরো কাজ বন্ধ হয়ে যায়। একটি মাত্র সার্ভারের উপর নির্ভরশীল।

✅ ডিস্ট্রিবিউটেড সিস্টেম (Git)

প্রতিটি ডেভেলপারের লোকাল কম্পিউটারে প্রোজেক্টের সম্পূর্ণ হিস্ট্রি থাকে। অফলাইনেও কাজ করা যায়, ইন্টারনেট পেলে পুশ করা যায়।

🐱 GitHub কী?

২০০৮ সালে Tom Preston-Werner সহ কয়েকজন ডেভেলপার মিলে GitHub ওয়েবসাইট তৈরি করেন। এটি Git রিপোজিটরিগুলোকে অনলাইনে হোস্ট করার সুবিধা দেয়। প্ল্যাটফর্মটি এতটাই জনপ্রিয় হয় যে ২০১৮ সালে Microsoft এটিকে ৭.৫ বিলিয়ন ডলারে অধিগ্রহণ করে।

02

গিট কীভাবে কাজ করে — চারটি স্টেজ

গিট মূলত ৪টি প্রধান স্টেজের মাধ্যমে কোড পরিচালনা করে। বিভিন্ন কমান্ড ব্যবহার করে ফাইলগুলোকে একটি স্টেজ থেকে অন্য স্টেজে নিয়ে যাওয়া হয়:

Working Directory
কোড লেখা ও এডিট
→
git add
Staging Area
কমিটের জন্য প্রস্তুত
→
git commit
Local Repo
লোকাল কম্পিউটারে সেভ
→
git push
Remote Repo
GitHub / GitLab
ওয়ার্কিং ডিরেক্টরি (Working Directory)

git init দিয়ে নতুন রিপোজিটরি তৈরি করলে বা git clone দিয়ে রিমোট থেকে ডাউনলোড করলে কোড এই স্টেজে থাকে। এখানেই মূলত কোড লেখা, এডিট করা বা নতুন ফাইল যোগ করা হয়।

স্টেজিং এরিয়া (Staging Area)

নির্দিষ্ট যে ফাইলগুলোর স্ন্যাপশট তৈরি করতে চান, সেগুলোকে git add কমান্ড দিয়ে স্টেজিং এরিয়াতে পাঠানো হয়। এর মানে আপনি গিটকে জানাচ্ছেন — এই পরিবর্তনগুলো সেভ করার জন্য প্রস্তুত।

লোকাল রিপোজিটরি (Local Repository)

স্টেজিং এরিয়ার ফাইলগুলোর একটি চেকপয়েন্ট বা স্ন্যাপশট তৈরি করতে git commit -m "message" ব্যবহার করা হয়। একটি উপযুক্ত মেসেজসহ কোডের পরিবর্তনগুলো পার্মানেন্টলি সেভ হয়।

রিমোট রিপোজিটরি (Remote Repository)

লোকালে সেভ করা কোড টিমের সাথে শেয়ার করতে git push কমান্ডে রিমোট সার্ভারে পাঠানো হয়। অন্যদের করা পরিবর্তন আনতে git pull ব্যবহৃত হয়।

03

গিট কনফিগারেশন ও ইনিশিয়ালাইজেশন

👤 প্রথমবার পরিচয় সেট করা

গিট ব্যবহারের শুরুতে আপনার নাম ও ইমেইল সেট করতে হবে, যাতে প্রতিটি কমিটে বোঝা যায় পরিবর্তনটি কে করেছে।

Terminal / Git Bash
# আপনার নাম সেট করুন git config --global user.name "Your Name" # আপনার ইমেইল সেট করুন git config --global user.email "your.email@example.com"

🚀 রিপোজিটরি তৈরি ও স্ট্যাটাস চেক

কমান্ড কাজ
git init একটি সাধারণ ফোল্ডারকে গিট রিপোজিটরিতে রূপান্তর করে। ভেতরে একটি লুকানো .git ডিরেক্টরি তৈরি হয়।
git status প্রোজেক্টের বর্তমান অবস্থা দেখায় — কোন ফাইল পরিবর্তন হয়েছে, কোনটি স্টেজিং এরিয়ায় আছে।
💡 প্রো টিপ

কাজের প্রতিটি ধাপে git status চালানো একটি ভালো অভ্যাস। এটি আপনাকে সবসময় প্রোজেক্টের বর্তমান অবস্থা সম্পর্কে সচেতন রাখবে।

04

স্টেজিং, কমিট ও হিস্ট্রি

📦 ফাইল স্টেজিং — git add

ওয়ার্কিং ডিরেক্টরি থেকে নির্দিষ্ট ফাইলগুলোকে কমিটের জন্য প্রস্তুত করতে স্টেজিং এরিয়ায় পাঠাতে হয়।

স্টেজিং কমান্ড
# নির্দিষ্ট ফাইল স্টেজ করুন git add filename.html # ফোল্ডারের সব ফাইল একসাথে স্টেজ করুন git add .

💾 কমিট করা — git commit

স্টেজিং এরিয়ার ফাইলগুলোর একটি স্ন্যাপশট বা চেকপয়েন্ট সেভ করাই হলো কমিট। প্রতিটি কমিটে একটি অর্থবহ মেসেজ দিতে হয়।

কমিট কমান্ড
git commit -m "Add user authentication"
📝 কমিট মেসেজের নিয়ম

কমিট মেসেজ সবসময় Present Tense বা বর্তমান কালে লিখুন। যেমন: "Fix bug" ✅ — "Fixed bug" ❌

↩️ ফাইল আনস্টেজ করা

ভুলক্রমে কোনো ফাইল স্টেজিং এরিয়ায় চলে গেলে তাকে আবার আনস্টেজ করতে:

আনস্টেজ কমান্ড
git restore --staged <filename>

📋 হিস্ট্রি দেখা — git log

কমান্ড কাজ
git log প্রোজেক্টের শুরু থেকে সকল কমিটের বিস্তারিত হিস্ট্রি দেখায় — কে, কখন, কী কমিট করেছে।
git log --oneline প্রতিটি কমিটকে সংক্ষেপে মাত্র এক লাইনে দেখায়। দ্রুত পর্যালোচনার জন্য অত্যন্ত কার্যকর।
05

ফাইল ইগনোর করা — .gitignore

🤔 কেন ফাইল ইগনোর করা দরকার?

প্রোজেক্টের সব ফাইল গিটহাবে পুশ করার প্রয়োজন হয় না। মূলত দুটি কারণে ফাইল ইগনোর করা হয়:

📁 বড় সাইজের ফাইল

ভারী বিল্ড ফাইল বা সফটওয়্যার সেটআপ (যেমন: ১ GB সাইজের .exe ফাইল)। এগুলো পুশ করলে অযথা ইন্টারনেট ও স্টোরেজ নষ্ট হয়।

🔒 গোপন তথ্য (Secrets)

.env ফাইলে থাকা ডেটাবেসের পাসওয়ার্ড বা API Key-এর মতো সিক্রেট ক্রেডেনশিয়াল। সিকিউরিটির কারণে এগুলো কখনোই গিটহাবে পুশ করা উচিত নয়।

⚙️ .gitignore কীভাবে কাজ করে?

  • প্রোজেক্টের রুট ফোল্ডারে .gitignore নামে একটি ফাইল তৈরি করতে হয়।
  • এই ফাইলের ভেতর যে ফাইল বা ফোল্ডারের নাম লেখা হয়, গিট সেগুলোকে সম্পূর্ণরূপে ইগনোর করে।
  • ইগনোর করা ফাইলগুলো লোকাল কম্পিউটারে ঠিকই থাকে এবং কাজও করে — কিন্তু গিট সেগুলোর পরিবর্তন ট্র্যাক করে না বা রিমোটে পুশ করে না।

✍️ .gitignore সিনট্যাক্স ও নিয়ম

.gitignore
# নির্দিষ্ট ফাইল ইগনোর .env secrets.json # পুরো ফোল্ডার ইগনোর (শেষে / দিতে হবে) builds/ node_modules/ # ফোল্ডারের নির্দিষ্ট ফাইল ইগনোর builds/build.txt
⚠️ গুরুত্বপূর্ণ

.gitignore ফাইলটিকেও অবশ্যই git add .gitignore দিয়ে স্টেজ এবং তারপর কমিট করে সেভ করতে হবে।

06

ফাঁকা ফোল্ডার ট্র্যাক করা — .gitkeep

⚠️ সমস্যা

গিট ডিফল্টভাবে কোনো ফাঁকা (empty) ডিরেক্টরি ট্র্যাক করে না। নতুন কোনো ফাঁকা ফোল্ডার তৈরি করলে git status চেক করলে দেখবেন গিট সেটিকে সম্পূর্ণ ইগনোর করছে।

✅ সমাধান

ফাঁকা ডিরেক্টরিকে ট্র্যাক করাতে চাইলে, সেই ফোল্ডারের ভেতরে .gitkeep নামের একটি ফাইল তৈরি করে রাখতে হয়। ফলে গিটের কাছে ফোল্ডারটি আর সম্পূর্ণ ফাঁকা থাকে না এবং সফলভাবে ট্র্যাক হতে শুরু করে। এরপর সাধারণ নিয়মে git add . ও git commit করে সেটিকে সেভ করা যায়।

07

ব্রাঞ্চিং — শাখা তৈরি ও ব্যবস্থাপনা

🌿 ব্রাঞ্চিং কী এবং কেন?

ব্রাঞ্চিং হলো মেইন ব্রাঞ্চের একটি সম্পূর্ণ স্বাধীন কপি তৈরি করা। গিট-এ ব্রাঞ্চ মূলত একটি নির্দিষ্ট কমিটকে পয়েন্ট করে রাখে।

সরাসরি মেইন ব্রাঞ্চে কাজ করলে মূল কোড নষ্ট বা ব্রেক হওয়ার ঝুঁকি থাকে। তাই নতুন ফিচার তৈরি বা পরীক্ষামূলক কাজের জন্য আলাদা ব্রাঞ্চ তৈরি করে সেখানে কাজ করা উচিত।

⌨️ ব্রাঞ্চিং কমান্ডসমূহ

কমান্ড কাজ
git branch <name> নতুন একটি ব্রাঞ্চ তৈরি করে।
git switch <name> এক ব্রাঞ্চ থেকে অন্য ব্রাঞ্চে সুইচ করে।
git switch -c <name> শর্টকাট — নতুন ব্রাঞ্চ তৈরি করে এবং সরাসরি সেই ব্রাঞ্চে সুইচ করে।
git merge <name> নির্দিষ্ট ব্রাঞ্চের কোড বর্তমান ব্রাঞ্চে যুক্ত (মার্জ) করে।
git branch -d <name> মার্জ হয়ে যাওয়া পুরোনো ব্রাঞ্চ ডিলিট করে।

🔥 ইমার্জেন্সি ফিক্স — ব্রাঞ্চের ব্যবহারিক সুবিধা

ফিচার ব্রাঞ্চে কাজ করার মাঝপথে হঠাৎ মেইন প্রোজেক্টে জরুরি বাগ ফিক্স করার প্রয়োজন হলে — অসম্পূর্ণ কাজ রেখেই git switch main দিয়ে মেইন ব্রাঞ্চে ফিরে যাওয়া যায়। ফিক্স করে কমিট করার পর আবার ফিচার ব্রাঞ্চে ফিরে অসম্পূর্ণ কাজ চালিয়ে নেওয়া যায়।

✨ ব্রাঞ্চিংয়ের বেস্ট প্র্যাকটিসেস

  • ডেডিকেটেড ব্রাঞ্চ: প্রতিটি নতুন ফিচার, বাগ ফিক্স বা পরিবর্তনের জন্য আলাদা ব্রাঞ্চ তৈরি করুন। সরাসরি মেইন ব্রাঞ্চে ডেভেলপমেন্ট করবেন না।
  • অর্থবহ নাম: ব্রাঞ্চের নাম পড়ে যেন এর কাজ বোঝা যায়। যেমন: feature/user-login বা bugfix/navbar-crash
  • শর্ট-লিভড: ব্রাঞ্চে কাজ দ্রুত শেষ করে মেইনে মার্জ করুন। দীর্ঘ সময় ফেলে রাখবেন না এবং মার্জের পর ডিলিট করে দিন।
08

মার্জ কনফ্লিক্ট ও সমাধান

💥 মার্জ কনফ্লিক্ট কী?

যখন দুটি ব্রাঞ্চে একই ফাইলের একই জায়গায় ভিন্ন ভিন্ন পরিবর্তন করা হয় এবং তারপর মার্জ করার চেষ্টা করা হয়, তখন গিট বুঝতে পারে না কোন পরিবর্তনটি রাখতে হবে। ফলে গিটের অটোমেটিক মার্জ প্রক্রিয়া ব্যর্থ হয় — একে মার্জ কনফ্লিক্ট বলে।

🔍 কনফ্লিক্ট মার্কার বোঝা

কনফ্লিক্ট দেখা দিলে গিট ফাইলের ভেতর কিছু মার্কার যুক্ত করে, যা দেখে সহজেই বোঝা যায় কোন অংশ কোন ব্রাঞ্চের:

<<<<<<< HEAD
বর্তমান ব্রাঞ্চের (মাস্টার) কোড এখানে
=======
ইনকামিং ব্রাঞ্চের (ফিচার) কোড এখানে
>>>>>>> feature-branch

🛠️ সমাধানের ধাপ

কনফ্লিক্ট ফাইল খুলুন

কনফ্লিক্ট হওয়া ফাইলটি কোড এডিটরে (যেমন: VS Code) খুলুন।

সিদ্ধান্ত নিন ও এডিট করুন

মার্কারগুলো মুছে ফেলুন এবং চূড়ান্ত রূপে কোন কোড রাখতে চান তা ঠিক করুন।

স্টেজ ও কমিট করুন

ফাইল সেভ করে git add . এবং git commit -m "Resolve merge conflict" চালান।

💡 VS Code-এর সুবিধা

VS Code-এ কনফ্লিক্ট দেখা দিলে সরাসরি "Accept Incoming Change" বা "Accept Current Change" বোতাম দেখায়, যা ক্লিক করে সহজেই কনফ্লিক্ট সমাধান করা যায়।

09

গিট স্ট্যাশিং — কাজ সাময়িকভাবে লুকিয়ে রাখা

📌 স্ট্যাশিং কী এবং কেন প্রয়োজন?

কাজের মাঝপথে অনেকগুলো ফাইলে অসম্পূর্ণ পরিবর্তন (Uncommitted changes) থাকা অবস্থায় হঠাৎ জরুরি ফিক্সের নির্দেশ আসলে — অসম্পূর্ণ কাজ কমিট করা সম্ভব হয় না। এই পরিস্থিতিতে অসম্পূর্ণ কাজগুলো সাময়িকভাবে একটি নিরাপদ জায়গায় লুকিয়ে রেখে ওয়ার্কিং ডিরেক্টরি ক্লিন করার জন্য স্ট্যাশিং ব্যবহার করা হয়।

⌨️ স্ট্যাশিং কমান্ডসমূহ

কমান্ড কাজ
git stash অসম্পূর্ণ পরিবর্তনগুলো সাময়িকভাবে স্ট্যাশে সেভ করে এবং ওয়ার্কিং ডিরেক্টরি ক্লিন করে। এরপর জরুরি কাজ করতে পারবেন।
git stash list স্ট্যাশে কী কী কাজ সেভ করা আছে তার তালিকা দেখায়।
git stash pop স্ট্যাশ করা কাজগুলো ওয়ার্কিং ডিরেক্টরিতে ফিরিয়ে আনে এবং স্ট্যাশ লিস্ট থেকে সেই এন্ট্রি মুছে ফেলে।
git stash apply স্ট্যাশ করা কাজ ফিরিয়ে আনে, কিন্তু স্ট্যাশ লিস্টে এন্ট্রি রেখে দেয়। (pop-এর বিপরীত)

git stash pop

কাজ ফেরত আনে ✅
স্ট্যাশ লিস্ট থেকে মুছে ফেলে ✅

git stash apply

কাজ ফেরত আনে ✅
স্ট্যাশ লিস্টে রেখে দেয় 🔄

10

গিট ট্যাগিং — ভার্সন চিহ্নিত করা

🏷️ ট্যাগিং কী?

গিট-এ নির্দিষ্ট কোনো কমিটকে বিশেষভাবে চিহ্নিত করতে বা সফটওয়্যারের কোনো নতুন ভার্সন/রিলিজ মার্ক করতে ট্যাগিং ব্যবহার করা হয়।

📦 ট্যাগের প্রকারভেদ

অ্যানোটেটেড ট্যাগ (Annotated)

মেসেজ ও বিস্তারিত তথ্যসহ তৈরি করা যায়। প্রোডাকশন রিলিজের জন্য উপযুক্ত।

git tag -a v1.0 -m "Release one"

লাইটওয়েট ট্যাগ (Lightweight)

শুধু ট্যাগের নাম দিয়ে তৈরি হয়, অতিরিক্ত কোনো মেসেজ বা তথ্য থাকে না।

git tag v1.1

👀 ট্যাগ দেখার উপায়

কমান্ড কাজ
git tag তৈরি করা সকল ট্যাগের তালিকা দেখায়।
git log --oneline হিস্ট্রিতে নির্দিষ্ট কমিটের পাশে ট্যাগ (যেমন: tag: v1.0) দেখায়।
11

গিট রিবেস — ক্লিন হিস্ট্রি

🔄 রিবেস কী এবং কেন প্রয়োজন?

সাধারণ মার্জে গিট একটি অতিরিক্ত "মার্জ কমিট" তৈরি করে। বারবার ব্রাঞ্চ তৈরি ও মার্জ করলে হিস্ট্রিতে এই মার্জ কমিটগুলো জমা হয়ে অগোছালো হয়ে যায়। এই অতিরিক্ত মার্জ কমিট এড়ানোর জন্যই রিবেস ব্যবহার করা হয়।

⚙️ রিবেস কীভাবে কাজ করে?

রিবেস ফিচার ব্রাঞ্চের কমিটগুলোকে টার্গেট ব্রাঞ্চের সর্বশেষ কমিটের ওপর এক এক করে পুনরায় সাজিয়ে বা "রিপ্লে" করে দেয়। ফলে হিস্ট্রি দেখে মনে হয় সব কাজ একটি ব্রাঞ্চেই ক্রমানুসারে করা হয়েছে — একদম ক্লিন ও লিনিয়ার।

রিবেস কমান্ড
# ফিচার ব্রাঞ্চে থাকা অবস্থায় রান করুন git rebase master

⚠️ মার্জ বনাম রিবেস

🔀 Merge

অরিজিনাল হিস্ট্রি সংরক্ষিত থাকে। অতিরিক্ত মার্জ কমিট তৈরি হয়।
কখন: অরিজিনাল হিস্ট্রি দেখতে চাইলে।

📏 Rebase

হিস্ট্রি ক্লিন ও লিনিয়ার রাখে। কমিট হ্যাশ পরিবর্তন হয়ে যায়।
কখন: পরিচ্ছন্ন হিস্ট্রি চাইলে।

🚫 সতর্কতা

রিবেস করলে পূর্বের কমিটগুলোর হ্যাশ (Hash) পরিবর্তন হয়ে যায়, অর্থাৎ এটি অরিজিনাল হিস্ট্রি নতুন করে লেখে (rewrite history)। শেয়ার করা বা পাবলিক ব্রাঞ্চে রিবেস করা থেকে বিরত থাকুন।

GitHub-এ কোড হোস্ট ও কোলাবোরেশন

☁️ GitHub কেন ব্যবহার করবেন?

  • লোকাল Git রিপোজিটরিগুলোকে অনলাইনে সুরক্ষিতভাবে হোস্ট বা সংরক্ষণ করা যায়।
  • একবার কোড গিটহাবে রাখলে ডিলিট না করা পর্যন্ত সারাজীবন সেখানে থাকে।
  • নিজের কাজ সমগ্র বিশ্বকে শোকেস করা যায় — পোর্টফোলিও হিসেবে কাজ করে।

📂 রিপোজিটরি তৈরি করা

New Repository তৈরি করুন

GitHub-এ "New Repository" অপশনে গিয়ে প্রোজেক্টের অর্থবহ নাম ও ডেসক্রিপশন দিন।

ভিজিবিলিটি সেট করুন

সবার জন্য উন্মুক্ত রাখতে Public, গোপন রাখতে Private সিলেক্ট করুন।

README ফাইল যোগ করুন

প্রোজেক্ট সম্পর্কে বিস্তারিত লিখে রাখতে একটি README (মার্কডাউন) ফাইল যুক্ত করুন।

🔑 ক্লোনিং ও SSH অথেনটিকেশন

ক্লোন ও পুশ কমান্ড
# GitHub থেকে রিপোজিটরি ক্লোন করুন git clone <repository-url> # কোড গিটহাবে পুশ করুন git push origin main
🔐 SSH Key সেটআপ

প্রাইভেট রিপোজিটরি ক্লোন বা পুশ করতে অথেনটিকেশন প্রয়োজন। লোকাল কম্পিউটারে একটি SSH Key তৈরি করে সেটি GitHub Settings → SSH and GPG Keys-এ যুক্ত করে দিন।

🔗 রিমোট কানেকশন ও সিঙ্ক

কমান্ড কাজ
git remote -v লোকাল রিপোজিটরি GitHub-এর কোন URL-এর সাথে যুক্ত তা দেখায়। ডিফল্ট নাম origin।
git push origin main লোকাল কমিট GitHub সার্ভারে পাঠায়।
git pull রিমোট থেকে নতুন পরিবর্তন নামিয়ে আনে এবং লোকাল কোডের সাথে মার্জ করে।
git fetch রিমোট থেকে পরিবর্তন নামিয়ে আনে, কিন্তু লোকালে মার্জ করে না।

git pull

ডাউনলোড ✅ + মার্জ ✅
একটি কমান্ডে সব কাজ

git fetch

শুধু ডাউনলোড ✅
মার্জ আলাদাভাবে করতে হবে
